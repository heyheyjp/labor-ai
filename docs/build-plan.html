Phase 2
Database
~1 day

Full schema DDL, migration tooling, and seed scripts for the static SOC taxonomy. The goal is a locally running Postgres with all 13 tables, correct FK constraints, and the full SOC hierarchy loaded — so the ingestion scripts in Phase 3 have a real target.

Deliverables
  • Alembic initialized, initial migration containing all 13 tables
  • SOC taxonomy seed script: loads soc_major_group → soc_minor_group → soc_broad_occupation → soc_detailed_occupation in insertion order
  • pgvector extension enabled in local Postgres and Supabase project
  • vector(1024) column on research_chunk with ivfflat index
  • All FK constraints and UNIQUE constraints verified with a smoke-test query
Constraints & indexes to verify
  • soc_detailed_occupation: UNIQUE(detailed_occupation_code, revision_year)
  • occupation_geo_estimate: UNIQUE(soc_detailed_occupation_id, geo_type, geo_code, survey_year)
  • occupation_ai_exposure: UNIQUE(soc_detailed_occupation_id, exposure_score_source)
  • app_user: UNIQUE(supabase_user_id), UNIQUE(github_user_id)
  • app_user_chat_message: CHECK(role IN ('user','assistant'))
  • Indexes on FK columns used in hot query paths
Done when
  • alembic upgrade head runs cleanly against local Postgres
  • SOC taxonomy seed script loads without errors
  • Query confirming ~830 rows in soc_detailed_occupation returns correct count
  • FK violation test: inserting an occupation_geo_estimate row with a non-existent soc_detailed_occupation_id raises a constraint error
  • Same migration runs cleanly against Supabase (staging)
Watch for
  • SOC 2018 vs SOC 2010 revision mismatches — Pew uses SOC 2018, some BLS files mix revisions. Confirm revision_year column is populated consistently.
  • pgvector version compatibility between local Docker image and Supabase's hosted version
Phase 3
Ingestion pipeline
2–3 days

Four standalone Python scripts that populate the operational tables, plus a fifth that pre-computes occupation name embeddings. This is the phase with the most unknowns — BLS files are finicky, SOC code matching between datasets requires care, and the embedding pipeline involves real API calls. Budget more time here than it looks like it needs.

⚠ Prerequisite — corpus must be assembled before Script 5 can run
  • The research corpus (PDFs/URLs) is not yet assembled. Sourcing, downloading, and cleaning the documents is a content task that must happen before the embedding pipeline can be validated end-to-end.
  • Target documents: WEF Future of Jobs 2025, Pew AI and the Future of Work 2023, Goldman Sachs AI Labor Market Report 2023, McKinsey The Future of Work After COVID-19, BLS OOH (selected pages), MIT Work of the Future reports, and any additional sourced PDFs.
  • Estimated effort: ~half a day. Do not treat as a background task — block Script 5 on it explicitly.
  • Scripts 1–4 (structured data) can proceed in parallel while corpus is being assembled.
Script 1 · BLS OEWS wages
  • Source: BLS OEWS 2024 national and state flat files (xlsx/csv)
  • Parse wage columns: A_PCT10, A_PCT25, A_MEDIAN, A_PCT75, A_PCT90, A_MEAN
  • Handle suppressed values — BLS uses *, #, ** codes for suppressed estimates. Store as NULL.
  • Match OCC_CODE to soc_detailed_occupation.detailed_occupation_code
  • Load national estimates first, then state, then metro
  • Target table: occupation_geo_estimate
Script 2 · BLS OOH projections
  • Source: BLS Employment Projections 2023–2033 table download
  • Fields: base employment, projected employment, numeric change, percent change, growth category string
  • SOC code matching — OOH uses a slightly different code format in some cases. Requires normalization.
  • Target table: occupation_projection
Script 3 · Pew AI exposure scores
  • Source: Pew Research Center AI Exposure 2023 dataset (CSV)
  • ~800+ occupations keyed by SOC code
  • Fields: exposure_score (0–1), automatable task fraction
  • Derive exposure_tier: high ≥ 0.7, medium 0.4–0.69, low < 0.4 — confirm thresholds against Pew methodology
  • SOC code matching: Pew uses SOC 2018. Confirm against soc_detailed_occupation.revision_year.
  • Target table: occupation_ai_exposure
Script 4 · Research corpus embedding
  • Source: curated PDFs and URLs (WEF, Goldman Sachs, Pew reports, BLS OOH pages, etc.) — see corpus prerequisite note above
  • Chunking strategy: ~500 token chunks with ~50 token overlap. Preserve source + title metadata.
  • Embed each chunk via Voyage AI (voyage-3, 1024 dimensions, input_type="document")
  • Insert to research_source + research_chunk with deduplication on source_url + chunk_index
  • Script should be idempotent — re-running skips already-embedded chunks
  • ~500–1,000 chunks expected. Budget ~$1–2 in Voyage API cost.
Script 5 · Occupation name embeddings
  • Embeds all ~830 occupation names via Voyage AI (voyage-3, input_type="query") and stores the vectors on soc_detailed_occupation
  • This eliminates the Voyage API call from the hot path of POST /analyses — initial analysis requests do a DB lookup instead of an API round-trip
  • Requires adding an embedding vector(1024) column to soc_detailed_occupation in the Phase 2 migration
  • Run after the SOC taxonomy seed is complete. Idempotent — skip rows that already have a non-null embedding
  • Total cost: ~830 embed calls. Negligible — under $0.01
  • Voyage is still called at query time when Claude invokes search_research on follow-ups — those query strings are arbitrary and cannot be pre-computed
Done when
  • All five scripts run to completion without errors against local Postgres
  • Row count checks: ~830 rows in occupation_geo_estimate (national), ~800 in occupation_ai_exposure, ~700 in occupation_projection
  • All ~830 rows in soc_detailed_occupation have a non-null embedding column
  • Spot-check: Software Developers (15-1252) has plausible wage, exposure, and projection data across all three tables
  • Vector similarity query against research_chunk returns relevant results for "AI impact on software development"
  • Vector lookup on soc_detailed_occupation.embedding by occupation ID returns a 1024-dim vector (confirms pre-compute ran correctly)
  • Scripts have run against Supabase (staging) and row counts match local
Known risks — budget extra time here
  • BLS column names differ between survey years and file formats. Validate headers before assuming field positions.
  • SOC code format inconsistencies across datasets. Some files omit the dash (151252 vs 15-1252). Normalize early.
  • Suppressed BLS values use multiple sentinel strings — test all known codes (*, #, **, †) before writing the NULL conversion.
  • Voyage API rate limits on bulk embedding. Add a small delay between batches.
  • Chunk size decisions affect retrieval quality. May need to experiment before committing.
Phase 4
Backend skeleton & auth
1–1.5 days

FastAPI app structure, auth dependency, rate limiter, and the two simpler endpoints fully implemented. At the end of this phase you have a running API that authenticates real Supabase users and returns real occupation data — a testable checkpoint before touching Claude.

App structure
  • app/main.py — FastAPI init, CORS middleware, router mounts
  • app/dependencies.py — get_current_user dependency (JWKS verification → app_user lookup)
  • app/db.py — async SQLAlchemy session factory
  • app/rate_limit.py — Upstash Redis client + per-user check function
  • app/models/ — SQLAlchemy ORM models mirroring schema
  • app/schemas/ — Pydantic request/response models
  • app/routers/ — one file per endpoint group
Endpoints implemented
  • GET /health — returns {"status": "ok"}
  • GET /occupations — auth required, queries soc_detailed_occupation joined to soc_major_group, sets Cache-Control and ETag headers
  • Auth middleware unit tests passing (from test plan Phase 1 suite)
  • Occupations endpoint tests passing
  • Railway deploy configured — both endpoints reachable at production URL
Done when
  • GET /health returns 200 in production
  • GET /occupations returns all ~830 occupations with correct shape and cache headers, authenticated with a real Supabase JWT
  • A request with no token or an invalid token returns 401
  • Auth and occupations test suites passing in CI
  • Upstash rate limiter returns 429 when limit is exceeded (verified in test, not manually)
Watch for
  • Supabase JWKS endpoint caching — fetch and cache the public key on startup, not on every request
  • Railway cold start latency — ensure the health check timeout is generous enough that Railway doesn't cycle the container on first deploy
  • Ingestion gate: all Phase 3 scripts must have been run against Supabase before POST /analyses will return meaningful data. This is a manual runbook prerequisite — it is not a Railway deploy hook or CI step. Do not deploy Phase 5 to production before confirming row counts in Supabase: ~830 in soc_detailed_occupation (with embeddings), ~800 in occupation_ai_exposure, ~700 in occupation_projection, and at least one successful vector similarity query on research_chunk.
Phase 5
Claude integration
2–2.5 days

The core of the system. Data fetch, vector search, context assembly, tool call execution, SSE streaming, output validation, and persistence — all in POST /analyses first, then POST /analyses/{id}/messages. Get it working non-streaming before adding streaming.

POST /analyses — build order
  • Step 1: Data fetch — query all four operational tables for the occupation in a single round-trip
  • Step 2: Vector search — retrieve pre-computed occupation embedding from soc_detailed_occupation.embedding (no Voyage call). Run cosine similarity query against research_chunk, join research_source
  • Step 3: Context assembly — build the structured prompt string from fetched data + research chunks
  • Step 4: Claude call (non-streaming) — tool definitions, round-trip 1 (tool calls), round-trip 2 (synthesis). Verify output shape.
  • Step 5: Output validation — run summary_json through Pydantic schema. Validate prose minimum length and refusal detection.
  • Step 6: Persistence — insert app_user_occupation_analysis row
  • Step 7: Convert to SSE streaming via FastAPI StreamingResponse
POST /analyses/{id}/messages
  • Ownership check (403 vs 404 distinction)
  • Server-side injection blocklist applied to message_text
  • Persist user message row before calling Claude
  • Condensed context assembly from analysis.summary_json
  • Fetch last 8 messages from app_user_chat_message
  • Claude call with tools available — same two-round-trip pattern
  • Stream response, persist assistant message with token counts
  • Send data event with message_id + token counts after stream completes
Claude integration details
  • Prompt caching: mark system prompt + tool definitions with cache_control
  • max_tokens: 2,048 for initial analysis, 1,024 for follow-ups
  • Model: claude-sonnet-4-20250514 — confirm latest available at build time
  • Tool descriptions are load-bearing — test that Claude routes query_database vs search_research correctly for both numeric and qualitative questions
  • Sentry error capture on all Anthropic API failures before surfacing 503 to client
SSE implementation notes
  • Use FastAPI's StreamingResponse with an async generator
  • Yield delta events as tokens arrive from the Anthropic streaming response
  • After stream completes: validate output, persist row, then yield data event, then done
  • On any exception mid-stream: yield error event with code + message, then close. Never leave the stream open on error.
  • Rate limit check happens before the generator opens — return a plain 429 HTTP response, not an SSE error event
Done when
  • Software Developers (15-1252) produces a complete, coherent streaming analysis end-to-end in production
  • summary_json in the persisted row matches expected schema
  • Follow-up question "What tasks are most at risk?" returns a relevant, streaming response
  • A follow-up question that warrants a fresh research search triggers search_research tool (verify via logs)
  • LLM output validation tests passing in CI (mocked Claude responses)
  • Injection defense tests passing in CI
  • Manual spot-check of 2–3 occupations from the test plan before moving to frontend
Watch for
  • Tool call routing errors — Claude may occasionally call the wrong tool or skip tools. System prompt ordering guidance mitigates this but test explicitly.
  • Streaming + persistence ordering — the row must not be inserted until the full response is received and validated. Don't persist partial output.
  • Context window overrun on follow-ups with long message histories — the 8-message cap should prevent this, but verify token counts in logs.
  • Voyage embedding latency adds to first-response time — log separately so you know whether to optimize.
Phase 6
Frontend
1–1.5 days

The API is fully working before any frontend work begins — no mocking required. The UI design is already settled from the mockup. This phase is primarily wiring existing designs to real data and handling the SSE stream lifecycle correctly.

Auth & session
  • Supabase Auth client initialized
  • GitHub OAuth sign-in flow — Supabase handles the redirect, stores the session, provides the JWT
  • JWT attached to all API requests via Axios/fetch interceptor or custom hook
  • Sign-out clears session and returns to auth screen
  • Auth state persisted across page refresh via Supabase session storage
Occupation search
  • Fetch GET /occupations on mount, store in module-level cache
  • Fuse.js instance initialized with the full occupation list
  • Debounced input → Fuse search → grouped dropdown results
  • Selection stores soc_detailed_occupation_id (UUID) for the analysis request
  • Empty input returns empty results (not the full 830-item list)
Analysis request & results
  • Submit POST /analyses, open EventSource connection
  • Accumulate delta events into prose string — stream renders progressively
  • On data event: store analysis_id, render charts from summary_json (AI exposure gauge, wage distribution bar chart, projection figure)
  • On done: close connection, clear loading state
  • On error: surface message to user, close connection, offer retry
  • 429 response (pre-stream): show rate limit message with retry timer
Follow-up chat
  • Follow-up input + suggestion chips rendered after analysis completes
  • Submit sends POST /analyses/{analysis_id}/messages
  • Same SSE stream pattern as analysis — accumulate deltas, render progressively
  • Client-side length gate: disable submit if input is empty or exceeds 2,000 chars
  • Conversation history rendered above the input
Done when
  • Full end-to-end flow works in production: sign in → search → analysis streams → charts render → follow-up question answered
  • Auth persists across page refresh
  • Rate limit message renders correctly (force a 429 in dev to verify)
  • SSE error event surfaces a readable message — not a blank screen or infinite spinner
  • Vitest suites for fuzzy search, SSE assembly, and client validation all passing
Watch for
  • EventSource does not support custom headers — the Supabase JWT cannot be sent via Authorization header in an EventSource request. Use a short-lived query parameter token or a cookie-based approach for the SSE endpoints.
  • Streaming render performance — long Claude responses updating React state on every delta token can cause jank. Throttle state updates if needed.
Phase 7
Tests & manual spot-checks
~1 day

A dedicated test pass fills gaps, cleans up tests written alongside Phase 5, and runs the full manual spot-check protocol from the test plan before anything is shown to anyone. Some tests should already exist from Phases 4 and 5 — this phase is for coverage gaps and the manual protocol.

Automated test gaps to fill
  • Input validation boundary cases (2,000 char limit, empty string, invalid UUID)
  • Ownership check distinction (403 vs 404) — if not already written in Phase 5
  • Rate limit pre-stream vs mid-stream behavior
  • Occupation response shape and cache headers
  • Frontend: Fuse.js edge cases (empty input, no match, grouping)
  • Frontend: SSE network drop handling
Manual protocol
  • Full spot-check of all 6 occupations from the test plan
  • All 5 injection attempt scenarios from the test plan
  • All 5 data edge case scenarios (suppressed data, missing Pew score, etc.)
  • Context window cutoff behavior — send 10+ messages, verify graceful handling
  • Full end-to-end flow on mobile viewport
Done when — system is demo-ready
  • All automated test suites passing in CI with no skips
  • All 6 spot-check occupations produce coherent, accurate analyses
  • No injection attempt produces a compliance response or off-topic output
  • Suppressed and missing data cases render gracefully in the UI
  • No console errors in the browser on any screen
  • Cold start to first analysis response measured and acceptable for a demo context
Note on test timing

LLM output validation and injection defense tests (from the test plan) are worth writing alongside Phase 5 while that code is fresh — not deferred entirely to this phase. Phase 7 is for coverage gaps, cleanup, and the manual protocol. Tests written late are better than no tests, but tests written alongside the code are better still.
