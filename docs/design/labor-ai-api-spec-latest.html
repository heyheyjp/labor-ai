Conventions

Base URL
All paths are relative to the Railway service root. e.g. https://labor-ai-api.up.railway.app. Configured via environment variable API_BASE_URL on the frontend.
Content type
All request bodies: application/json. Non-streaming responses: application/json. Streaming responses: text/event-stream.
IDs
All IDs are UUIDs in string form. e.g. "3fa85f64-5717-4562-b3fc-2c963f66afa6".
Timestamps
All timestamps returned as ISO 8601 UTC strings. e.g. "2026-03-09T14:32:00.000Z".
Validation errors
FastAPI returns 422 Unprocessable Entity with Pydantic's error detail for malformed request bodies — field name, location, and message. These are distinct from application-level 400 errors.
CORS
FastAPI CORS middleware allows requests from the Vercel frontend origin only. Configured via environment variable ALLOWED_ORIGIN.

Authentication

All endpoints except GET /health require a valid Supabase-issued JWT in the Authorization header.

Request header
Authorization: Bearer <supabase_jwt>

FastAPI uses a get_current_user dependency declared once and injected into each protected endpoint. The dependency:

Dependency flow
  • Extracts the JWT from the Authorization header. Returns 401 if absent or malformed.
  • Verifies the JWT signature against Supabase's public JWKS endpoint (https://<project>.supabase.co/auth/v1/.well-known/jwks.json). Returns 401 if invalid or expired.
  • Extracts sub claim (Supabase user ID) from the verified payload.
  • Looks up the corresponding app_user row by supabase_user_id. Returns 401 if not found — this would indicate a user who exists in Supabase Auth but has not yet completed first-login provisioning.
  • Returns the resolved app_user row to the endpoint. Downstream code never touches the raw JWT.

Error responses

All error responses share a consistent JSON shape. SSE error events use the same message field.

Error response shape
{ "error": { "code": "RATE_LIMIT_EXCEEDED", // machine-readable constant "message": "Query limit reached. Try again in 42 minutes." } }
StatusCode constantWhen
400 BAD_REQUEST Request is structurally valid (passes Pydantic) but semantically wrong — e.g. soc_detailed_occupation_id not found in the database.
401 UNAUTHORIZED Missing, expired, or invalid JWT. Also returned if app_user row cannot be resolved from the token.
403 FORBIDDEN The analysis exists but belongs to a different user. Ownership check failed.
404 NOT_FOUND analysis_id path parameter does not exist in app_user_occupation_analysis.
422 VALIDATION_ERROR FastAPI/Pydantic validation failure — missing required field, wrong type, constraint violation (e.g. message_text exceeds 2,000 chars). Returns Pydantic's full error detail.
429 RATE_LIMIT_EXCEEDED Per-user Upstash rate limit hit. Response includes retry_after_seconds in the error body.
500 INTERNAL_ERROR Unhandled exception. Sentry captures the full trace. Message is deliberately vague to avoid leaking internals.
503 UPSTREAM_UNAVAILABLE Anthropic API or Supabase returned an error or timed out. Retryable — message indicates which upstream.

SSE streaming

The two POST endpoints respond with Content-Type: text/event-stream via FastAPI's StreamingResponse. The connection stays open until the done or error event is sent.

Each SSE message follows the standard format: a named event line and a data line containing a JSON string.

SSE event types
event: delta data: {"text": "Software Developers occupy a nuanced..."} event: data data: {"analysis_id": "3fa85f64-...", "summary_json": {...}} event: done data: {} event: error data: {"code": "UPSTREAM_UNAVAILABLE", "message": "Anthropic API timed out."}
Event semantics
  • delta — one text chunk from Claude's streaming response. Accumulate these to build the full prose string. Sent throughout generation.
  • data — sent once, immediately after the last delta. Contains the persisted record IDs and structured payload for chart rendering. On POST /analyses this includes analysis_id and summary_json. On POST /analyses/{id}/messages this includes message_id.
  • done — stream is complete. Frontend closes the EventSource connection.
  • error — something went wrong mid-stream. Frontend should surface the message to the user and close the connection. Sentry has already captured the trace on the backend.
Rate limit timing

The Upstash rate limit check happens before the stream opens. A 429 is returned as a normal HTTP error response (not an SSE event) so the frontend can handle it without opening an EventSource at all.

GET /occupations Auth required

Returns the full list of ~830 detailed occupations from soc_detailed_occupation, joined to soc_major_group for grouping in the dropdown UI. This response is cached — the occupation list is static and loaded once from BLS. The frontend uses Fuse.js to run fuzzy search client-side against this list; there are no per-keystroke network calls.

Response headers
HeaderValueNotes
Cache-Control public, max-age=86400 24-hour client cache. The dataset changes only when a new BLS SOC revision is loaded — rare enough that a manual cache bust (via deploy) is acceptable.
ETag generated Hash of the response body. Enables 304 Not Modified on repeat requests within the cache window.
Response · 200 OK
application/json
{ "occupations": [ { "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6", "detailed_occupation_code": "15-1252", "detailed_occupation_name": "Software Developers", "major_group_name": "Computer and Mathematical Occupations" }, // ... ~830 records total ] }
Client-side search
  • The frontend initialises a Fuse.js instance with this list on mount, keying on detailed_occupation_name.
  • Search runs in-memory on every debounced input event — no further API calls until the user selects an occupation.
  • Results are grouped by major_group_name for the dropdown's grouped display.
  • The id (UUID) is what gets submitted to POST /analyses — not the SOC code string.
POST /analyses Auth required SSE stream

Creates a new occupation analysis session. Fetches structured data from Supabase, runs a vector similarity search against the research corpus, calls the Anthropic API (tool call round-trip + synthesis), persists the result to app_user_occupation_analysis, and streams Claude's prose response back to the client token by token. The structured data payload is sent as a single data event once generation is complete.

Request body
FieldTypeNotes
soc_detailed_occupation_id req string · UUID The id of the selected occupation from soc_detailed_occupation. Validated against the database — returns 400 if not found.
Server-side sequence
  • 1. Auth. Resolve app_user from JWT via dependency injection.
  • 2. Validate. Confirm soc_detailed_occupation_id exists. Return 400 if not.
  • 3. Rate limit. Increment per-user counter in Upstash. Return 429 if exceeded — before opening the stream.
  • 4. Data fetch. Query Supabase in a single round-trip: occupation_geo_estimate (national wages), occupation_ai_exposure, occupation_projection, occupation_task (top automatable tasks by importance score).
  • 5. Vector search. Embed the occupation name via Voyage AI. Run cosine similarity search against research_chunk. Retrieve top 4 chunks + join research_source for citation metadata.
  • 6. Claude call — round 1. Send system prompt + structured data + tool definitions. Claude returns tool_use blocks if it needs additional data. Execute any tool calls locally and send results back.
  • 7. Claude call — round 2. Claude synthesizes prose. Stream delta events to the client as tokens arrive.
  • 8. Persist. After the stream completes, insert app_user_occupation_analysis row with summary_text, summary_json, and model_name.
  • 9. data event. Send the data SSE event with analysis_id and summary_json for chart rendering.
  • 10. done event. Close the stream.

text/event-stream

POST /analyses/{analysis_id}/messages Auth required SSE stream

Appends a user message to an analysis session and streams Claude's response. Validates that the analysis belongs to the requesting user. Persists both the user message and the assistant reply to app_user_chat_message. Claude receives condensed structured context plus the last 8 messages of conversation history, with tools available for fresh data retrieval if the question warrants it.

Path parameters
ParameterTypeNotes
analysis_id req string · UUID ID of the app_user_occupation_analysis session. Returns 404 if not found, 403 if it belongs to a different user.
Request body
FieldTypeNotes
message_text req string User's follow-up question. Max 2,000 characters — enforced by Pydantic, returns 422 if exceeded. Stored to app_user_chat_message after server-side validation and blocklist check.
Server-side sequence
  • 1. Auth. Resolve app_user from JWT.
  • 2. Ownership check. Fetch app_user_occupation_analysis by analysis_id. Return 404 if not found. Return 403 if app_user_id does not match the requesting user.
  • 3. Input validation. Server-side blocklist check on message_text for prompt injection patterns. Reject with 400 if flagged.
  • 4. Rate limit. Increment per-user counter in Upstash. Return 429 if exceeded.
  • 5. Persist user message. Insert app_user_chat_message row with role='user'.
  • 6. Assemble context. Build condensed occupation context from analysis.summary_json. Fetch last 8 messages from app_user_chat_message ordered by created_at_utc ASC.
  • 7. Claude call — round 1. Send system prompt + condensed context + conversation history + tool definitions. Execute any tool calls Claude requests.
  • 8. Claude call — round 2. Stream delta events to client.
  • 9. Persist assistant message. Insert app_user_chat_message row with role='assistant', model_name, prompt_tokens, output_tokens.
  • 10. data event. Send message_id of the persisted assistant reply.
  • 11. done event. Close the stream.
SSE stream · data event (after generation)
text/event-stream
event: data data: { "message_id": "8d3b1c2a-4e5f-6789-abcd-ef0123456789", "prompt_tokens": 842, "output_tokens": 317 }
Context window discipline

Only the last 8 messages are sent to Claude on each follow-up call — older messages are not summarized in V1, simply excluded. This caps per-request token cost as conversations grow long. If Claude needs to reference something from much earlier in the conversation, the user should restate it.

Claude Integration
