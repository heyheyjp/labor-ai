Observability Plan

What to instrument, what to log, and what to alert on across the frontend, backend, and ingestion pipeline. Scoped to the real needs of a demo-grade system — not enterprise theater.

v1 · March 2026
Sentry · Railway · Vercel
Sentry
Errors + performance

Single dashboard covering both frontend (Browser SDK) and backend (Python SDK). Free tier is sufficient — 5k errors/month, 10k performance transactions.

Railway
Infra metrics + logs

Built-in CPU, memory, and request metrics for the FastAPI container. Stdout logs available in the Railway dashboard. No additional setup required.

Vercel
Frontend analytics + logs

Build logs, function logs (if any), and Web Vitals available out of the box. Sufficient for a frontend with no server-side rendering complexity.

Token budget tracking

Token usage per user is already persisted in app_user_chat_message. Operational queries against this table provide cost visibility without additional tooling.

Frontend
Layer 1
Frontend

The frontend's primary observability concern is SSE stream failures and auth errors — the two places where things can go wrong silently and leave the user staring at a spinner. Sentry handles both. Vercel provides everything else for free.

Sentry Browser SDK
  • Initialize in app/layout.tsx (or equivalent root). Configure DSN via environment variable.
  • Automatic capture of unhandled JS errors and unhandled promise rejections
  • Manual capture at SSE error event and onerror handler — include analysis_id and occupation name as context
  • Manual capture on any fetch that returns 5xx — include endpoint, status code, and response body if available
  • Set user.id on the Sentry scope after successful auth — links errors to specific sessions without storing PII
  • Source maps uploaded at build time so stack traces are readable
What Vercel gives you for free
  • Build logs — every deploy, full output, failures surface immediately
  • Web Vitals — LCP, FID, CLS across real user sessions. Useful for catching streaming render jank.
  • Function logs — relevant if you add any Next.js API routes or middleware
  • Preview deploy URLs — every branch gets a unique URL, making it easy to test before merging
Key events to capture manually
  • SSE stream error — Sentry.captureException with {analysis_id, occupation_name, error_code}
  • SSE network drop — EventSource onerror without a prior error event. Distinct from a server-sent error.
  • Auth failure on API call — 401 response from any endpoint after the user is supposedly signed in. Indicates a token expiry or session sync issue.
  • Rate limit hit — 429 response. Not an error exactly, but useful to know if it's happening frequently.
What not to send to Sentry from the frontend
  • User message text — never attach message_text to Sentry events. It's free-text user input and could contain anything.
  • The full SSE accumulated prose — large payloads slow down Sentry ingestion and eat quota.
  • JWT or session tokens — obvious, but worth stating explicitly given they're in scope on the frontend.
Backend
Layer 2
Backend

The backend has more moving parts and more failure modes. Sentry captures exceptions automatically. The more valuable investment is structured logging — making it easy to reconstruct what happened in a given request after the fact, without instrumenting every function.

Sentry Python SDK
  • Initialize in app/main.py before the app starts. FastAPI integration is built-in — all unhandled exceptions are captured automatically with request context.
  • Manually call sentry_sdk.capture_exception() in the except block for Anthropic API failures and Supabase failures before returning 503. Auto-capture would work too but explicit capture lets you attach context.
  • Use Sentry's with sentry_sdk.push_scope() as scope to attach analysis_id, occupation_id, and app_user_id to events — makes triaging much faster.
  • Performance tracing: enable on POST /analyses and POST /analyses/{id}/messages to measure end-to-end latency including Anthropic round-trips. Sample rate 1.0 is fine at demo scale.
What Railway gives you for free
  • CPU and memory metrics with 7-day history — enough to spot a memory leak or runaway process
  • Request count and error rate at the container level
  • Stdout log streaming — all print() and logger output visible in the Railway dashboard in near-real time
  • Deploy history — easy rollback if something goes wrong post-deploy
  • Automatic container restart on crash — with a notification if it happens repeatedly

Structured logging

Use Python's standard logging module with a JSON formatter (e.g. python-json-logger). Every log line is a JSON object — Railway can display it as structured data and it's trivial to grep. Key fields on every log line: level, timestamp, request_id, app_user_id, endpoint.

Example log lines — POST /analyses request lifecycle
{"level": "INFO", "event": "analysis_requested", "app_user_id": "uuid", "occupation_id": "uuid"} {"level": "INFO", "event": "data_fetch_complete", "duration_ms": 42} {"level": "INFO", "event": "vector_search_complete", "chunks_returned": 4, "duration_ms": 118} {"level": "INFO", "event": "claude_round1_complete", "tools_called": ["search_research"], "duration_ms": 1840} {"level": "INFO", "event": "claude_stream_complete", "prompt_tokens": 1204, "output_tokens": 487, "duration_ms": 6230} {"level": "INFO", "event": "analysis_persisted", "analysis_id": "uuid"} {"level": "ERROR", "event": "anthropic_timeout", "duration_ms": 30000, "error": "ReadTimeout"} {"level": "WARN", "event": "rate_limit_hit", "app_user_id": "uuid", "retry_after_s": 2520} {"level": "WARN", "event": "output_validation_failed","reason": "exposure_score_out_of_range", "value": 1.8}
Always log
  • Request start and end for streaming endpoints — with total duration
  • Each major step duration: data fetch, vector search, Claude round 1, Claude round 2
  • Tool calls Claude made in round 1 — which tools, in what order
  • Token counts from every Claude response (prompt_tokens, output_tokens)
  • Output validation failures with the specific reason and offending value
  • Rate limit hits with app_user_id and retry window
  • Any upstream error (Anthropic, Supabase, Voyage, Upstash) with status code and duration
Never log
  • User message text — free-text input, treat as sensitive
  • The assembled prompt string — contains user input and research chunk text
  • Claude's prose response — large and potentially contains echoed user content
  • JWT tokens or any credential
  • The token query parameter if you use one for SSE auth — sanitize it from Railway's request logs

Token & cost tracking

Token counts are already persisted in app_user_chat_message.prompt_tokens and app_user_chat_message.output_tokens. No additional tooling required — a handful of SQL queries give you complete cost visibility.

Total token spend this month
SELECT SUM(prompt_tokens) AS total_prompt, SUM(output_tokens) AS total_output, COUNT(*) AS total_messages FROM app_user_chat_message WHERE created_at_utc >= date_trunc('month', now()) AND role = 'assistant';
Top users by token consumption
SELECT app_user_id, SUM(prompt_tokens + output_tokens) AS total_tokens, COUNT(*) AS message_count FROM app_user_chat_message WHERE role = 'assistant' AND created_at_utc >= now() - interval '30 days' GROUP BY app_user_id ORDER BY total_tokens DESC LIMIT 20;
Rough cost reference (Claude Sonnet, March 2026)

Verify current pricing at anthropic.com/pricing before relying on these figures. At typical rates, a single initial analysis (prompt ~1,200 tokens, output ~500 tokens) costs on the order of $0.003–0.006. A follow-up message (~800 prompt, ~300 output) is roughly $0.002–0.004. At demo scale — a handful of users, tens of analyses — total monthly Claude spend should be well under $5. The Upstash rate limiter is the safeguard if usage unexpectedly spikes.

Ingestion pipeline
Layer 3
Ingestion pipeline

The ingestion scripts are run once (or occasionally when new data is available) — not a long-running service. Full observability infrastructure is not warranted. Good stdout logging, row count verification, and idempotency checking are sufficient. Sentry is not needed here.

Logging approach
  • Print progress to stdout at meaningful checkpoints — not every row, but every batch or file section
  • Log row counts before and after each insert batch so you can spot partial loads
  • Log suppressed value counts — how many BLS cells were set to NULL due to suppression codes. If this number is unexpectedly high it signals a parsing problem.
  • Log SOC code match failures — occupations in the source data that couldn't be matched to soc_detailed_occupation. A non-zero count needs investigation.
  • Log Voyage API batch progress and any rate limit backoffs
  • Final summary line for each script: rows inserted, rows skipped (duplicates), rows failed
Verification queries
  • Run after each script. Confirm row counts match expectations before moving to the next script.
  • SELECT COUNT(*) FROM occupation_geo_estimate WHERE geo_type = 'national' — expect ~830
  • SELECT COUNT(*) FROM occupation_ai_exposure — expect ~800
  • SELECT COUNT(*) FROM occupation_projection — expect ~700
  • SELECT COUNT(*) FROM research_chunk WHERE embedding IS NOT NULL — expect 500–1,000
  • Spot-check query: all four tables populated for SOC 15-1252 (Software Developers) with plausible values
Example script output — BLS OEWS loader
[oews_loader] Starting BLS OEWS 2024 national load [oews_loader] Parsed 832 occupation rows from oews_national_M2024_dl.xlsx [oews_loader] SOC match failures: 3 # investigate these [oews_loader] Suppressed wage values set to NULL: 214 # normal — ~25% suppression rate is expected [oews_loader] Inserted: 829 | Skipped (duplicate): 0 | Failed: 0 [oews_loader] State files: processing 51 files... [oews_loader] State load complete. Inserted: 41,832 rows [oews_loader] Done in 14.2s
Idempotency is the key property
  • Every script should be safe to re-run without duplicating data. Use INSERT ... ON CONFLICT DO NOTHING on the UNIQUE constraints.
  • The embedding script should check whether a chunk already has a non-null embedding before calling Voyage — avoids re-paying for already-embedded content.
  • Log skipped rows separately from inserted rows so you know the re-run worked correctly.
Watch for in logs
  • SOC match failure count > 10 — likely a code format normalization problem
  • Suppressed value count much higher or lower than expected — signals a parsing issue with BLS sentinel codes
  • Embedding script finishing with fewer chunks than expected — could indicate chunking logic cutting off early
  • Any Voyage API error that wasn't retried — chunks without embeddings will silently degrade RAG quality
Alerts
Operations
Alerts worth setting up

At demo scale, the alert surface should be small. The goal is to find out about real problems — not to generate noise. These are all configurable in Sentry's free tier or Railway's notification settings.

AlertWhere to configureSeverityWhy it matters
New error type in Sentry Sentry — "first seen" alert high Any previously unseen exception class is worth immediate attention. Not every instance — just the first occurrence of a new type.
Anthropic API error rate > 0 Sentry — issue alert on anthropic_timeout / anthropic_error events high The Anthropic API going down or rate-limiting you breaks the core feature entirely. Want to know immediately.
Railway container restart Railway — notification settings high An unexpected restart indicates a crash or OOM. The app recovers automatically but you want to know it happened.
Output validation failure Sentry — alert on output_validation_failed event medium Claude returned structurally invalid data. Rare but important — could indicate a model change or prompt regression.
Rate limit hits > 5/hour Sentry — alert on rate_limit_hit event count threshold low A few hits are expected. A spike suggests either a legitimate heavy user or an abuse pattern worth investigating.
Supabase connection failure Sentry — alert on DB connection error events high Every endpoint except health depends on the database. A connection failure is a full outage.
Dashboards
If & when to build them
Dashboards

Four distinct audiences, each caring about different things. None of these need to be built for V1 — Sentry's built-in views and the token-tracking SQL queries cover what's actually needed at demo scale. The cost/usage dashboard is the first one worth building if real users appear. The LLM quality dashboard is the most interesting but also the hardest to make genuinely useful without investing in proper evals.

Operational health — is the system working?
  • Error rate — 5xx responses per hour across all endpoints. A spike here is the most important signal in the system.
  • Anthropic API success rate — tracked separately from general error rate. Anthropic going down is a full feature outage; it deserves its own line.
  • Latency p50 / p95 on POST /analyses and POST /analyses/{id}/messages — the two endpoints users are waiting on. Time-to-first-token is the more user-relevant metric for streaming.
  • Active SSE connections — how many analyses are in flight right now.
  • Rate limit hit rate — a spike suggests either heavy legitimate use or an abuse pattern.
  • Railway's built-in dashboard covers CPU and memory. Sentry's issues dashboard covers errors. Together they're most of this view with no custom work.
Cost & usage — what is this costing?
  • Daily token spend — total prompt + output tokens per day, with a cost estimate. The single most important number if this scales.
  • Token trend — rolling 30-day chart. Flat is fine; a sudden slope change needs explanation.
  • Prompt vs output token split — output tokens cost more. A widening output share means longer Claude responses, which may or may not be intentional.
  • Top users by token consumption — the 80/20 distribution is almost always steep. Knowing who the heavy users are is essential for rate limit tuning.
  • Average tokens per analysis vs per follow-up — useful for understanding where cost is actually coming from.
  • Entirely queryable from app_user_chat_message. The first one worth promoting to an actual dashboard if usage grows.
LLM quality — is Claude producing good output?
  • Output validation failure rate — how often does summary_json fail schema validation? Should be near zero; any nonzero rate after deploy deserves investigation.
  • Prose length distribution — histogram of summary_text character counts. A cluster of very short responses indicates refusals or errors that slipped past validation.
  • Tool call distribution — which tools is Claude choosing, and how often? A sudden shift may indicate a prompt regression.
  • Recent validation failures — a table showing the last 20 failures with occupation, reason, and offending value. More actionable than a rate metric alone.
  • None of these tell you whether an analysis is accurate — only that it's structurally correct. Accuracy requires evals, which are out of scope for V1.
User behavior — what are people doing?
  • Most searched occupations — which SOC codes are generating the most analyses. Useful for understanding what resonates and for prioritizing spot-checks.
  • Session depth — distribution of follow-up message counts per session. Are users reading and leaving, or engaging with the conversation?
  • Occupations with highest follow-up rate — often a signal of either high interest or confusing/incomplete initial output.
  • Daily active users — distinct app_user_ids with at least one analysis in the last 24 hours.
  • Drop-off by screen — what fraction of users who authenticate go on to run an analysis? The data is already in the DB.
Build order if you get there
  • First: Cost & usage — a few SQL queries promoted to a Retool or Metabase view. Highest practical value, lowest build cost. Directly informs rate limit and budget decisions.
  • Second: Operational health — Sentry already covers most of this. The missing piece is time-to-first-token latency, which requires adding a timestamp to the structured log at stream open vs first delta event.
  • Third: User behavior — interesting once you have enough users to make the patterns meaningful. Not worth building for single-digit users.
  • Fourth: LLM quality — the validation failure table is easy and high value. The rest requires real evals infrastructure to be genuinely useful rather than just reassuring.
Scope
Scope
What not to bother with

Observability infrastructure has a real setup and maintenance cost. These are all legitimate tools for production systems at scale — they're just not worth the investment here.

Out of scope for V1
  • Distributed tracing (OpenTelemetry, Datadog APM, Honeycomb) — valuable when you have many services. For one FastAPI service the Sentry performance tracing is sufficient.
  • Custom metrics pipeline (Prometheus, Grafana) — Railway's built-in metrics cover CPU/memory. Custom business metrics at this scale don't need a dedicated pipeline.
  • Log aggregation (Datadog Logs, Papertrail, Logtail) — Railway's log streaming is sufficient. A log aggregation service makes sense when you need search across weeks of logs or multiple services.
  • Uptime monitoring (Better Uptime, Pingdom) — reasonable eventually, but for a demo the Railway health check restart behavior is sufficient.
  • LLM evals pipeline — automated scoring of analysis quality against a benchmark set. The right long-term investment, but manual spot-checks cover V1.
  • Real user session recording (LogRocket, FullStory) — overkill for a single-feature demo app.
Upgrade path if this goes beyond a demo
  • Add Logtail or Papertrail if you need searchable log history beyond Railway's retention window
  • Add uptime monitoring (Better Uptime free tier) if real users depend on availability
  • Promote the token-usage SQL queries to a simple dashboard (Retool, Metabase) if cost tracking becomes a regular concern
  • Add an LLM evals pipeline if the prompt or model changes frequently and you need regression detection
