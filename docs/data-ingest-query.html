Ingestion · Script 0
SOC taxonomy seed

The Standard Occupational Classification (SOC) hierarchy is the backbone of the data model. All operational tables FK into soc_detailed_occupation. The taxonomy is loaded once before any other ingestion script runs, in strict insertion order from top to bottom of the hierarchy so FK constraints are satisfied.

Source & structure
  • Source: BLS SOC 2018 classification files (CSV downloads from bls.gov)
  • Four levels: major group (23) → minor group (~100) → broad occupation (~450) → detailed occupation (~830)
  • Each level has a code string and a title. Codes follow a structured pattern: 15-0000 (major), 15-1200 (minor), 15-1250 (broad), 15-1252 (detailed)
  • revision_year set to 2018 throughout. Future SOC revisions can coexist in the same tables via the UNIQUE(code, revision_year) constraint.
Insertion order
  • 1. soc_major_group — 23 rows, no FK dependencies
  • 2. soc_minor_group — FK → soc_major_group
  • 3. soc_broad_occupation — FK → soc_minor_group
  • 4. soc_detailed_occupation — FK → soc_broad_occupation
  • All subsequent ingestion scripts reference soc_detailed_occupation.id (UUID) by looking up detailed_occupation_code. The UUID is the operational join key throughout.
Core seed logic (Python)
def seed_soc_taxonomy(session: Session) -> None: # Load each level from BLS CSV files # INSERT ... Ingestion · Script 1
BLS OEWS wages

The Occupational Employment and Wage Statistics (OEWS) survey provides wage distributions and employment counts by occupation and geography. The 2024 release covers national estimates, 50 states + DC, and ~600 metropolitan areas. All three geo levels are loaded into a single table keyed by (soc_detailed_occupation_id, geo_type, geo_code, survey_year).

Source files
  • oews_national_M2024_dl.xlsx — national estimates, one row per occupation
  • oews_state_M2024_dl.xlsx — state-level estimates, one row per occupation × state
  • oews_metro_M2024_dl.xlsx — MSA-level estimates, one row per occupation × metro area
  • All three files share the same column schema. Load national first, then state, then metro.
  • Source columns relevant to the model: OCC_CODE, OCC_TITLE, TOT_EMP, A_MEAN, A_MEDIAN, A_PCT10, A_PCT25, A_PCT75, A_PCT90
Target table fields
  • soc_detailed_occupation_id — looked up by normalized OCC_CODE
  • geo_type — 'national', 'state', or 'metro'
  • geo_code — FIPS state code or MSA code; 'US' for national
  • geo_name — human-readable name (e.g. 'California', 'San Francisco-Oakland-Hayward, CA')
  • est_worker_count — from TOT_EMP
  • annual_mean_wage_usd, annual_median_wage_usd — from A_MEAN, A_MEDIAN
  • annual_wage_p10/p25/p75/p90 — from corresponding A_PCT* columns
  • survey_year — 2024 for this load
BLS suppression handling

BLS suppresses estimates when the sample is too small to produce a reliable figure, or when publication would allow identification of a specific employer. Suppressed cells are indicated by sentinel strings rather than empty values. All sentinels must be converted to NULL before insert — a missed sentinel stored as a string in a numeric column will cause a type error or, worse, silently succeed if the column is text.

Known suppression sentinel values
  • * — estimate not released (most common)
  • # — wage exceeded the top-coded value ($239,200)
  • ** — employment estimate not released
  • † — data not available for this occupation/area
  • Empty string "" — treated as suppressed in some file versions
Suppression normalization
SUPPRESSED = {"*", "**", "#", "†", ""} def parse_wage(raw: str) -> float | None: if raw.strip() in SUPPRESSED: return None return float(raw.replace(",", ""))

V1 scope: national estimates only

The frontend displays national wage distributions only. State and metro files are loaded into the database for completeness and future use, but the application query always filters on geo_type = 'national'. Loading all three geo levels adds ~42,000 rows but costs nothing in query performance since the national filter is indexed.

Ingestion · Script 2
BLS OOH projections

The BLS Employment Projections program publishes 10-year outlook data for detailed occupations. The 2023–2033 release is the current dataset. Unlike the OEWS files, the projections table is not exhaustive — not every detailed occupation has a projection, and some projections cover a broader occupation grouping rather than a single SOC code.

Source & fields
  • Source: BLS Employment Projections 2023–2033 table (CSV download from bls.gov/emp)
  • OCC_CODE — SOC code. Requires the same normalization as OEWS.
  • EMPL_BASE — base year (2023) employment in thousands
  • EMPL_PROJ — projected year (2033) employment in thousands
  • EMPL_CHNG_NUMER — numeric change (thousands)
  • EMPL_CHNG_PCT — percent change
  • CATEG_CHANGE — BLS categorical label: "Much faster than average", "Faster than average", "As fast as average", "Slower than average", "Little or no change", "Decline"
Target table fields
  • soc_detailed_occupation_id — FK to soc_detailed_occupation
  • base_year — 2023
  • projection_year — 2033
  • base_employment — raw count (multiply BLS thousands by 1,000)
  • projected_employment — raw count
  • change_abs — absolute change in headcount
  • change_pct — percent change as a float (e.g. 25.9)
  • growth_category — BLS categorical string stored verbatim
Coverage gap

Not all ~830 detailed occupations have projections. The BLS sometimes groups several detailed occupations under a single projected category, or excludes occupations with insufficient data. The occupation_projection table will have ~700 rows — ~130 occupations will have no projection data. The application and Claude's system prompt must handle NULL projection gracefully rather than treating absence as an error.

Ingestion · Script 3
Pew AI exposure scores

The Pew Research Center's 2023 AI exposure study scored ~800 occupations on a 0–1 scale based on the degree to which AI could perform or assist with the tasks comprising each job. It also identified which specific tasks within each occupation are automatable. This is the primary AI risk signal in the system.

Source & fields
  • Source: Pew Research Center dataset download (CSV). Available via the report's methodology appendix.
  • Keyed by SOC 2018 code — matches soc_detailed_occupation.revision_year = 2018 directly
  • ai_exposure_score — float 0–1. Higher = more exposed.
  • pct_tasks_automatable — fraction of job tasks Pew classified as AI-automatable
  • No per-task breakdown in the main dataset — task-level data comes from O*NET (out of scope for initial V1 load)
Tier derivation
  • exposure_tier is derived at load time from the score, not stored in the Pew source
  • high — score ≥ 0.70
  • medium — score 0.40–0.69
  • low — score < 0.40
  • Thresholds are based on Pew's own methodology notes. Confirm against the report before finalizing.
  • exposure_score_source field set to 'pew_2023' — allows future sources (e.g. a 2025 update) to coexist via the UNIQUE(occupation_id, source) constraint
Load logic
def derive_tier(score: float) -> str: if score >= 0.70: return "high" if score >= 0.40: return "medium" return "low" def load_pew_exposure(session: Session, filepath: str) -> LoadResult: df = pd.read_csv(filepath) for _, row in df.iterrows(): occupation_id = get_occupation_id(session, row["soc_code"]) if not occupation_id: continue session.execute( insert(OccupationAiExposure).values( soc_detailed_occupation_id = occupation_id, exposure_score = float(row["ai_exposure_score"]), exposure_tier = derive_tier(float(row["ai_exposure_score"])), tasks_automatable_pct = float(row["pct_tasks_automatable"]), exposure_score_source = "pew_2023", ).on_conflict_do_nothing() ) session.commit()
Ingestion · Script 4
Research corpus & embeddings

The research corpus is the qualitative layer of the system — the body of reports, studies, and articles from which Claude draws cited evidence. Each document is chunked into overlapping passages, each passage is embedded via Voyage AI into a 1,024-dimensional vector, and both the text and the vector are stored in Postgres. This enables semantic similarity search at query time: given an occupation, retrieve the passages most relevant to it.

Document preparation
Corpus contents (initial)
  • WEF Future of Jobs 2025
  • Pew AI and the Future of Work 2023
  • Goldman Sachs AI Labor Market Report 2023
  • McKinsey The Future of Work After COVID-19
  • BLS Occupational Outlook Handbook (selected pages)
  • MIT Work of the Future reports
  • Additional sourced PDFs — TBD at build time
Chunking strategy
  • Target chunk size: ~500 tokens. At this size, each chunk is semantically coherent — a paragraph or two — without being so long that it dilutes the embedding signal.
  • Overlap: ~50 tokens between adjacent chunks. Prevents a sentence that straddles a chunk boundary from being effectively invisible to retrieval.
  • Chunk boundaries respect sentence endings — do not split mid-sentence.
  • Title and source metadata prepended to each chunk text before embedding, so the vector reflects context not just content.

import voyageai voyage = voyageai.Client() # reads VOYAGE_API_KEY from environment def embed_and_store( session: Session, source_id: UUID, chunks: list[str], batch_size: int = 100 ) -> None: for i in range(0, len(chunks), batch_size): batch = chunks[i : i + batch_size] # Voyage embed call — model="voyage-3" returns 1024-dim vectors result = voyage.embed(batch, model="voyage-3") vectors = result.embeddings # list[list[float]], len == len(batch) for j, (text, vector) in enumerate(zip(batch, vectors)): chunk_index = i + j session.execute( insert(ResearchChunk) .values( research_source_id = source_id, chunk_index = chunk_index, chunk_text = text, embedding = vector, # pgvector stores list[float] natively ) .on_conflict_do_nothing() # idempotent on (source_id, chunk_index) ) session.commit() time.sleep(0.5) # conservative rate limit buffer
research_source row
  • source_url — canonical URL or file path. UNIQUE — deduplication key.
  • source_title — human-readable title used in citations
  • source_type — 'report', 'article', 'government', etc.
  • published_date — date of publication, used in citation display
research_chunk row
  • research_source_id — FK to research_source
  • chunk_index — 0-based position within the document
  • chunk_text — the raw passage text, stored verbatim for inclusion in prompts
  • embedding — vector(1024), pgvector type. Cosine similarity index (ivfflat) on this column.

Occupation name embeddings

All ~830 occupation names are embedded once at ingestion time and stored on soc_detailed_occupation. This eliminates a Voyage API round-trip from the hot path of POST /analyses — initial analysis requests retrieve the pre-computed vector directly from Postgres instead of calling the Voyage API live. Voyage is still called at query time when Claude invokes the search_research tool on follow-up messages, since those query strings are arbitrary and cannot be pre-computed.

Query time
Query path 1
Structured data fetch

When a user submits an occupation for analysis, FastAPI executes a single Postgres round-trip that retrieves all structured data needed for the prompt: wages, exposure score, employment projection, and top automatable tasks. This happens before any call to Claude.

Single-round-trip data fetch (SQLAlchemy)
async def fetch_occupation_data( session: AsyncSession, occupation_id: UUID ) -> OccupationData: # Wages — national only for V1 wages = await session.execute( select(OccupationGeoEstimate) .where(OccupationGeoEstimate.soc_detailed_occupation_id == occupation_id) .where(OccupationGeoEstimate.geo_type == "national") .where(OccupationGeoEstimate.survey_year == 2024) ).scalar_one_or_none() # Exposure exposure = await session.execute( select(OccupationAiExposure) .where(OccupationAiExposure.soc_detailed_occupation_id == occupation_id) .where(OccupationAiExposure.exposure_score_source == "pew_2023") ).scalar_one_or_none() # Projection projection = await session.execute( select(OccupationProjection) .where(OccupationProjection.soc_detailed_occupation_id == occupation_id) .where(OccupationProjection.base_year == 2023) ).scalar_one_or_none() # Top automatable tasks, ordered by importance tasks = await session.execute( select(OccupationTask) .where(OccupationTask.soc_detailed_occupation_id == occupation_id) .where(OccupationTask.is_automatable == True) .order_by(OccupationTask.importance_score.desc()) .limit(5) ).scalars().all() return OccupationData(wages=wages, exposure=exposure, projection=projection, tasks=tasks)
NULL handling contract
  • Any of the four result objects may be None — occupations with no Pew score, no projection data, or suppressed wages are all valid states.
  • The context assembly step (see below) is responsible for formatting None values into prompt text that Claude can reason about gracefully — e.g. "Employment projection: not available for this occupation."
  • summary_json mirrors these nulls in its structure. The frontend renders null fields with appropriate fallback UI — empty chart slots with a "data unavailable" label, not errors.
Query path 2
Vector similarity search

After the structured data fetch, FastAPI retrieves the pre-computed occupation embedding from soc_detailed_occupation.embedding and runs a cosine similarity search against research_chunk to find the four most semantically relevant research passages. No Voyage API call is needed on the initial analysis path. Voyage is only called live when Claude invokes the search_research tool on follow-up messages, where the query string is arbitrary and can't be pre-computed.

pgvector index
  • Index type: ivfflat with lists=100 — appropriate for a corpus of 500–2,000 chunks. Trades a small amount of recall for faster search.
  • For a corpus this size the performance difference vs exact search is negligible, but ivfflat is the correct habit for when the corpus grows.
  • Index is on research_chunk(embedding vector_cosine_ops)
  • Created in the Alembic migration after the pgvector extension is enabled.
Query text composition
  • The query text passed to Voyage is not just the bare occupation name. A richer query produces better retrieval: "AI impact and automation risk for {occupation_name}"
  • On search_research tool calls from Claude during follow-ups, the query text is Claude's own formulation — passed through as-is from the tool input. This is intentional: Claude's query reflects what it actually needs, which is often more specific than a generic occupation query.
  • Query text is never logged — it may reflect user message content in follow-up contexts.
Query path 3
Context assembly

After the structured fetch and vector search complete, FastAPI assembles the Claude prompt. The structured data and research chunks are formatted into a human-readable plaintext block — not JSON. Plaintext is more token-efficient and Claude reads it more reliably than deeply nested JSON for prompt inputs.

Why plaintext over JSON for prompt context
  • JSON structural characters ({}, [], "", commas) consume tokens without adding meaning. A plaintext wage line like median: $132,000 is roughly half the tokens of its JSON equivalent.
  • Claude reads and reasons about structured plaintext at least as reliably as JSON for prompt inputs — this is distinct from tool call outputs, where JSON is the correct format.
  • Plaintext is easier to read during development and debugging. You can print the assembled context and immediately see what Claude sees.
  • The summary_json payload (stored in the DB, sent in the SSE data event) is still JSON — it needs to be machine-parseable on the frontend for chart rendering. The distinction is prompt input vs structured output.

Voyage AI integration

Voyage AI is an embedding model provider. Its role in this system is narrow and well-defined: convert text to 1,024-dimensional vectors. It is called at ingestion time (once, for the entire corpus) and at query time (once per analysis request, for the occupation query). It never handles structured data, user PII, or conversation content.

Integration points
  • Ingestion (once): scripts/embed_corpus.py — embeds all research chunks in batches. ~500–1,000 API calls total across all documents. Cost: ~$1–2.
  • Query time (per analysis): app/services/vector_search.py — embeds the occupation query string. One API call, <100ms, fraction of a cent.
  • Tool call (conditional): When Claude invokes search_research during a follow-up, FastAPI calls Voyage to embed Claude's query string before running the pgvector search. Same code path as the initial analysis query.
  • Voyage is never called for: user message text, structured data, JWT validation, or anything outside the two contexts above.
Model & configuration
  • Model: voyage-3 — current flagship model, 1,024 dimensions, strong retrieval benchmark performance
  • Input type: document at ingestion time, query at search time — Voyage optimizes the embedding differently for each. Always specify the correct input type.
  • Batch size: 100 at ingestion. The Voyage API accepts up to 128 texts per call; 100 is a conservative default.
  • Auth: VOYAGE_API_KEY environment variable. Read by the client on initialization.
  • Rate limit: Voyage free tier is sufficient for corpus ingestion with a 500ms delay between batches. Upgrade if corpus grows significantly.
Client initialization and correct input_type usage
import voyageai # Initialize once at module level — client reads VOYAGE_API_KEY from env voyage_client = voyageai.Client() # At ingestion: use input_type="document" # Voyage optimizes document embeddings for storage and retrieval def embed_documents(texts: list[str]) -> list[list[float]]: result = voyage_client.embed(texts, model="voyage-3", input_type="document") return result.embeddings # At query time: use input_type="query" # Voyage optimizes query embeddings for similarity matching against stored documents def embed_query(text: str) -> list[float]: result = voyage_client.embed([text], model="voyage-3", input_type="query") return result.embeddings[0] # The asymmetric input_type is intentional and important. # Documents and queries are embedded into different subspaces # optimized for retrieval — using "document" for both degrades recall.
Dimension consistency is critical

Every vector in research_chunk.embedding must be produced by the same model (voyage-3, 1,024 dimensions) as the query-time embedding. If you switch models or change the embedding dimension, all stored vectors become incomparable with new query vectors — you must re-embed the entire corpus. The model name is not stored in the database in V1; treat it as a configuration constant and document it here.

SOC code matching

All four ingestion sources use SOC codes as their occupation identifier, but they do not all format them identically. Code normalization must happen before any database lookup — a mismatch here silently drops rows rather than raising an error.

Known format variations
  • Canonical format (used in the database): 15-1252 — two digits, hyphen, four digits
  • BLS OEWS files sometimes emit 151252 — no hyphen. Normalize by inserting hyphen at position 2.
  • BLS OOH projection files occasionally include a trailing zero for broad-level codes used as detailed proxies: 15-1250. These may not match a soc_detailed_occupation row — log as a miss rather than failing silently.
  • Pew dataset uses the canonical format consistently — least likely to cause issues.
  • Leading/trailing whitespace in CSV cells. Always .strip() before lookup.
SOC normalization function
def normalize_soc_code(raw: str) -> str: code = raw.strip() # Insert hyphen if missing: "151252" → "15-1252" if len(code) == 6 and code.isdigit(): code = f"{code[:2]}-{code[2:]}" # Validate expected format if not re.match(r"^\d{2}-\d{4}$", code): raise ValueError(f"Unexpected SOC format: {raw!r}") return code
SOC revision mismatch
  • The database stores revision_year = 2018 throughout. BLS OEWS 2024 uses SOC 2018. Pew 2023 uses SOC 2018. These are consistent.
  • Some BLS projection files and O*NET data still reference SOC 2010 codes for certain occupations. If you encounter codes with no match in the database, check whether a SOC 2010 → 2018 crosswalk is needed. BLS publishes a crosswalk file for this purpose.
  • In V1, unmatched codes are logged and skipped — not failed. A post-ingestion log review should confirm the miss count is small (<10) and not systematic.
