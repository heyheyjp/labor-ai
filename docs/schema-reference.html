Naming conventions

Table names: singular, snake_case
occupation_task not occupation_tasks — follows the original relational model; a table names a type, not a collection
Be explicit — no ambiguous abbreviations
detailed_occupation_code not soc_code; major_group_name not major_group; soc_detailed_occupation_id not occupation_id
Currency: append _usd to mean and median wage fields only
annual_mean_wage_usd, annual_median_wage_usd — percentile fields omit _usd as currency is established by context
Wage time period: always explicit
annual_mean_wage_usd not mean_wage
Percentile fields: p{N} format
annual_wage_p10, annual_wage_p90 — distinct from pct which is reserved for percentage
Percentage fields: _pct suffix
tasks_automatable_pct — never use p{N} format for percentages
Boolean fields: is_ prefix
is_automatable not automatable
Timestamps (datetime with time zone): _utc suffix, always stored in UTC
created_at_utc, updated_at_utc on mutable tables. Immutable tables (app_user_occupation_analysis, app_user_chat_message) carry created_at_utc only — omitting updated_at_utc enforces write-once semantics at the schema level.
Dates (calendar date, no time component): no _utc suffix
published_date — a date has no timezone
Surrogate PKs: UUID on all tables that need them, field named id
Used where no meaningful natural key exists, or where the natural key alone can't guarantee uniqueness across data versions (e.g. SOC revision cycles)
FK field names: {table_name}_id for UUID surrogate FKs
e.g. soc_detailed_occupation_id on operational tables, soc_major_group_id on soc_minor_group — the _id suffix signals a UUID FK. String natural keys use descriptive names like detailed_occupation_code.
Application layer tables: app_user_ prefix
Tables that belong to the application's user-facing data model — as opposed to the reference/operational data from external sources — are prefixed app_user_ where they are user-scoped. app_user itself is the root; app_user_occupation_analysis and app_user_chat_message extend it.
Immutable records: no updated_at_utc
Records that must never be edited after insert — analysis outputs, chat messages — omit updated_at_utc. The absence of the field signals intent. If a message is wrong, a new record is appended; old records are never mutated.

Operational Tables — populated by ingestion scripts
occupation_geo_estimate
Employment counts and wage estimates by occupation and geography. One row per occupation × geography × year.
Source: BLS OEWS (annual)

BLS surveys ~1.1 million employers annually. These are estimates derived from a survey, not a census — hence the table name. BLS suppresses values where sample size is too small, so many fields are nullable. Geography is stored inline (not normalized) — geo entities carry no independent attributes worth a separate table at this stage.

FieldTypeSourceNotes
idPK UUID generated Surrogate key. Logical key is the composite unique constraint below.
soc_detailed_occupation_idFK str BLS OEWS References soc_detailed_occupation. Aggregate/group-level rows (codes ending in 0, 00, 0000) are filtered out during ingestion — only detailed occupation rows are loaded.
survey_yearreq int BLS OEWS Year of the OEWS survey. Latest available: 2024. Storing year enables multi-year accumulation for trend queries without schema changes.
geo_typereq str derived "national" | "state" | "metro". BLS publishes three separate files per year — this field merges them into one table.
geo_nameopt str | null BLS OEWS Human-readable geography. e.g. "California" or "San Francisco-Oakland-Hayward, CA". Null for national rows.
geo_codeopt str | null BLS OEWS Machine-readable geography code. FIPS for states (e.g. "06" = California), MSA code for metro areas. Null for national. Note: state FIPS and MSA codes come from different numbering systems and are not globally unique — always filter by geo_type alongside geo_code.
est_worker_countopt int | null BLS OEWS Estimated number of employed workers in this occupation and geography. Nullable — BLS suppresses small-sample estimates.
annual_mean_wage_usdopt float | null BLS OEWS Mean annual wage. Can be skewed upward by top earners — median is usually more representative.
annual_median_wage_usdopt float | null BLS OEWS Median annual wage (50th percentile). Most commonly cited figure. "Half of workers in this occupation earn more than $X."
annual_wage_p10opt float | null BLS OEWS 10th percentile annual wage. Proxy for entry-level / early-career compensation floor.
annual_wage_p25opt float | null BLS OEWS 25th percentile annual wage.
annual_wage_p75opt float | null BLS OEWS 75th percentile annual wage.
annual_wage_p90opt float | null BLS OEWS 90th percentile annual wage. Together the four percentile columns let a user roughly self-locate in their occupation's wage distribution.
created_at_utcauto datetime generated Auto-set on insert.
updated_at_utcauto datetime generated Auto-set on insert and update.
Constraints
  • UNIQUE (soc_detailed_occupation_id, geo_type, geo_code, survey_year)
Notes
  • Geo fields are inline — no separate reference table. Geography has no rich independent attributes to warrant normalization at this stage.
  • Ingestion is append-only by year — old year rows are retained, enabling trend queries over time.
  • BLS also publishes hourly wage figures in the same files — not included in V1 but naming would follow hourly_mean_wage_usd etc. if added later.
occupation_ai_exposure
AI automation exposure scores per occupation. One row per occupation per scoring methodology.
Source: Pew Research Center (derived from O*NET task data)

Pew evaluated each O*NET work activity for AI automatability and rolled scores up to the occupation level. The surrogate PK + unique constraint on (code, source) allows future methodologies (Goldman, OECD, etc.) to coexist as additional rows rather than replacing existing data.

FieldTypeSourceNotes
idPK UUID generated Surrogate key. Logical key is (soc_detailed_occupation_id, exposure_score_source).
soc_detailed_occupation_idFK str Pew / O*NET References soc_detailed_occupation.
exposure_score_sourcereq str metadata Identifies the methodology that produced the score. Value: "pew_2023" for all current data. Part of the unique constraint — allows multiple methodologies per occupation in future.
exposure_scorereq float Pew Normalized 0.0–1.0. Higher = more exposed. e.g. Data Entry Keyers ≈ 0.9, EMTs ≈ 0.15.
exposure_tierreq str derived "high" | "medium" | "low". Bucketed from exposure_score at ingestion time.
tasks_automatable_pctreq float Pew / O*NET Percentage of this occupation's tasks Pew classified as highly automatable. e.g. 0.72 = 72% of tasks. More interpretable than the normalized score for end users.
exposure_summaryopt str | null generated Plain English explanation of why this occupation has this score. Generated once via Claude API after initial ingestion and stored permanently. Null until generation script is run.
created_at_utcauto datetime generated Auto-set on insert.
updated_at_utcauto datetime generated Auto-set on insert and update.
Constraints
  • UNIQUE (soc_detailed_occupation_id, exposure_score_source)

occupation_task
Individual work tasks per occupation, with AI automation risk per task. ~10–30 rows per occupation.
Source: O*NET task database + Pew automation classifications

Where occupation_ai_exposure gives a single score per occupation, this table provides the underlying task-level breakdown — which specific activities within the job are automatable. Enables "which parts of my job are most at risk?" queries.

FieldTypeSourceNotes
idPK UUID generated Surrogate key.
soc_detailed_occupation_idFK str O*NET References soc_detailed_occupation. Many-to-one — each occupation has many tasks.
task_descriptionreq str O*NET The actual task text. e.g. "Write code to create applications or programs." Directly surfaceable to users.
task_typereq str O*NET "core" = performed by most workers in the occupation. "supplemental" = performed by some. Core tasks are most relevant for exposure analysis.
is_automatablereq bool Pew Pew's binary classification — is this task highly exposed to AI automation?
automation_risk_levelreq str derived More granular than the boolean: "high" | "medium" | "low". Derived from Pew's underlying scores where available.
importance_scoreopt float | null O*NET O*NET's importance rating for this task, 0–5 scale. Higher = more central to the job. Enables weighting exposure by task importance rather than just counting tasks.

updated_at_utcauto datetime generated Auto-set on insert and update.

Holds all document-level metadata. The deduplication check on ingestion happens here — look up source_url before processing a document. If it already exists, skip. Not occupation-specific — any labor market or AI displacement research belongs here.

research_chunk
Chunked text with vector embeddings. Child of research_source. The RAG retrieval unit.
Source: chunked + embedded at ingestion time

Each source document is split into overlapping chunks (~500 tokens, ~100 token overlap), embedded via Voyage AI, and stored here. At query time the user's question is embedded and the most similar chunks are retrieved via cosine similarity and fed to Claude.

FieldTypeSourceNotes
idPK UUID generated Surrogate key.
research_source_idFK UUID generated References research_source.id. All chunks for a document share the same research_source_id.
chunk_indexreq int generated 0-indexed position of this chunk within its source document. Allows reconstructing document order and fetching adjacent chunks when more context is needed.
chunk_textreq str ingestion The actual text content (~400–600 tokens). Overlaps slightly with adjacent chunks to avoid cutting ideas mid-sentence. This is what Claude reads when a chunk is retrieved.
embeddingopt vector(1024) Voyage AI 1024-dimensional float vector from Voyage AI's voyage-3 model. Stored in pgvector's native Vector column. Used for cosine similarity search. Null until embedding script is run. Dimension is model-specific — document in a comment if the model changes.
created_at_utcauto datetime generated Auto-set on insert.
updated_at_utcauto datetime generated Auto-set on insert and update.
Constraints
  • UNIQUE (research_source_id, chunk_index)
Application Layer — user, session, and messaging tables
app_user
One row per authenticated user. Application-level profile extending Supabase's internal auth.users.
Source: created on first GitHub OAuth sign-in via Supabase Auth

Supabase Auth brokers the GitHub OAuth flow and creates a row in its internal auth.users table. This table is the application's own profile record, linked to auth.users via supabase_user_id. FastAPI validates the Supabase-issued JWT on every request and resolves the caller to an app_user row. All user-scoped application data FKs here, not to auth.users directly.

FieldTypeSourceNotes
idPK UUID generated Surrogate key. Used as FK target by all application-layer tables.
supabase_user_idreq UUID Supabase Auth References auth.users.id in Supabase's internal schema. The link between this application profile and the Supabase-managed auth record. Unique — one app_user per Supabase auth identity.
github_user_idreq bigint GitHub OAuth GitHub's immutable integer user ID. This is the stable, deduplication-safe identifier — it never changes even if the user renames their account. bigint not text; GitHub IDs are integers.
github_loginreq text GitHub OAuth GitHub username at time of last sign-in. e.g. "jpratt". Display only — mutable, GitHub users can rename themselves. Never use as a join key. Updated on each successful sign-in to stay current.
created_at_utcauto timestamptz generated Auto-set on insert. Records when the user first signed in.
updated_at_utcauto timestamptz generated Auto-set on insert and update. Updated on each sign-in so github_login can be refreshed if the user has renamed their account.
Constraints
  • UNIQUE (supabase_user_id)
  • UNIQUE (github_user_id)
Auth flow
  • Supabase Auth handles the GitHub OAuth dance. FastAPI receives a Supabase JWT in the Authorization header and verifies it using Supabase's public JWKS endpoint — no Auth.js, no separate OAuth library needed.
  • On first sign-in: Supabase creates auth.users row → FastAPI creates app_user row. On subsequent sign-ins: FastAPI upserts github_login in case of rename.
  • Scope requested from GitHub: read:user only. No email, no repo access.
app_user_occupation_analysis
One row per user × occupation analysis session. The session container for all subsequent chat messages.
Source: generated by FastAPI + Claude on occupation selection

Created when a user selects an occupation and Claude synthesizes the initial analysis. Functions simultaneously as a document (holds the generated analysis) and a session container (all app_user_chat_message rows for this conversation FK here). Records are immutable after insert — no updated_at_utc.

FieldTypeSourceNotes
idPK UUID generated Surrogate key. Referenced by app_user_chat_message.app_user_occupation_analysis_id.
app_user_idFKreq UUID application References app_user.id. Required — all analysis sessions are user-scoped in V1.
soc_detailed_occupation_idFKreq UUID application References soc_detailed_occupation.id. The occupation this analysis is for.
summary_textreq text Claude API The prose Claude generated — what the user reads. Plain text with inline citation markers e.g. [1], [2]. Stored as-generated; never edited.
summary_jsonreq jsonb application Structured data payload assembled for this analysis: Pew exposure score and tier, wage percentiles, employment projection, top automatable tasks, citation metadata. Enables re-rendering charts without re-querying the underlying tables. Schema is application-defined and may evolve.
model_namereq text application The Claude model string used to generate this analysis. e.g. "claude-sonnet-4-5". Stored for auditability — useful if model is upgraded and you want to identify analyses generated by older versions.
created_at_utcauto timestamptz generated Auto-set on insert. No updated_at_utc — this record is immutable after creation.
Design notes
  • No updated_at_utc: the record is write-once. If analysis needs to be regenerated (e.g. model upgrade), insert a new row. Old rows are retained for audit and comparison.
  • A user can have multiple analysis rows for the same occupation — each represents a distinct session. No unique constraint on (app_user_id, soc_detailed_occupation_id).
  • "When was this session last active?" requires SELECT MAX(created_at_utc) FROM app_user_chat_message WHERE app_user_occupation_analysis_id = ?. Acceptable at this scale.
app_user_chat_message
One row per message turn within an analysis session. Append-only.
Source: user input + Claude API responses

Stores every message turn in a follow-up conversation — both user messages and assistant replies. Always attached to an app_user_occupation_analysis session. Records are immutable: no edits, no deletes. If a response must be retried, a new row is appended. Ordered by created_at_utc within a session.

FieldTypeSourceNotes
idPK UUID generated Surrogate key.
app_user_idFKreq UUID application References app_user.id. Denormalized from the parent analysis for efficient "fetch all messages for user" queries without an extra join.
app_user_occupation_analysis_idFKreq UUID application References app_user_occupation_analysis.id. Groups all messages belonging to one analysis session. Required — messages are always session-scoped.
soc_detailed_occupation_idFKreq UUID application References soc_detailed_occupation.id. Denormalized from the parent analysis. Makes "fetch all messages for occupation X" a single-table query — a hot path worth the redundancy.
rolereq text application CHECK constraint: 'user' | 'assistant'. Mirrors the Claude API's message role vocabulary exactly.
message_textreq text user / Claude API The message content. For role='user': the raw input after server-side validation and sanitization (max 2,000 chars enforced at API layer). For role='assistant': Claude's response text.
model_nameopt text | null application Null for role='user' messages. For role='assistant': the Claude model string used. e.g. "claude-sonnet-4-5".
prompt_tokensopt int | null Claude API Null for role='user'. For role='assistant': input token count from the Claude API response. Used to compute per-user token consumption for budget enforcement.
output_tokensopt int | null Claude API Null for role='user'. For role='assistant': output token count. Combined with prompt_tokens for total consumption: SUM(prompt_tokens + output_tokens) per user per period.
created_at_utcauto timestamptz generated Auto-set on insert. The canonical ordering field within a session — ORDER BY created_at_utc ASC. No updated_at_utc: records are immutable.
Constraints
  • CHECK (role IN ('user', 'assistant'))
Design notes
  • No updated_at_utc: append-only by design. A message is never edited. A failed or unwanted assistant response is handled by inserting a new pair, not updating the old one.
  • No message_index: created_at_utc with millisecond precision is sufficient for ordering sequential chat turns. Revisit if pagination or guaranteed ordinal display becomes necessary.
  • No response_to_message_id: conversation is linear. A tree structure adds complexity with no V1 benefit.
  • Token budget enforcement: SELECT SUM(prompt_tokens + output_tokens) FROM app_user_chat_message WHERE app_user_id = ? AND created_at_utc > [period_start]. FastAPI checks this before each Claude call and returns 429 if over budget.

Security & token limiting

Layer 1 — Vercel edge
IP-based rate limiting before requests reach the application. First line of defense against scrapers and bots. No app code involved.
Layer 2 — Supabase JWT validation
FastAPI rejects any request without a valid Supabase-issued JWT. Verified against Supabase's public JWKS endpoint. Gates all data access behind authentication.
Layer 3 — Upstash rate limiting
Per-user query budget keyed by GitHub user ID. TTL resets hourly. FastAPI returns 429 before Claude is called. Handles legitimate over-use, not bad actors.
Layer 4 — Pydantic input validation
Every FastAPI endpoint declares a strict schema. message_text: max 2,000 chars. Occupation search input: length-bounded and stripped. Malformed payloads rejected before business logic.
Layer 5 — Prompt injection defense
User input only ever appears in the 'user' turn of the Claude API call. System prompt is fully server-controlled and never interpolated with user-supplied strings. This is the single most critical rule.
Layer 6 — Claude max_tokens cap
Hard max_tokens on every Claude API call: 2,048 for initial analysis, 1,024 for follow-ups. Prevents runaway generation cost from any single request.
Layer 7 — Context window discipline
Follow-up calls send last 6–8 messages only, not full history. Older turns are not summarized in V1 — simply truncated. Caps per-request token cost as conversations grow.
Layer 8 — Monthly token budget
SUM(prompt_tokens + output_tokens) per user per calendar month queried from app_user_chat_message before each Claude call. Hard block (429) if over threshold. Threshold TBD based on cost targets.
Layer 9 — Input blocklist (V1)
Server-side check for known prompt injection patterns ("ignore previous instructions", "you are now", etc.) applied to message_text before prompt assembly. Lightweight V1 approach — replace with Claude-as-classifier in V2 if needed.
